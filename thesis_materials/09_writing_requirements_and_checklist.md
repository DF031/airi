# 写作规范与检查清单

本文件结合项目本身和本地《人工智能系本科毕业设计论文写作培训》PDF 提炼，用于写作阶段自检。

## 写作原则

### 1. 先搭树，再写叶子

论文不是从第一句话顺着写到最后一句，而是先确定树状结构：

- 题目：一句话说明研究对象。
- 摘要：约 10 句话说明问题、方法、系统、实验和结论。
- 章节：约 50 个小节句说明完整逻辑。
- 段落：每段先有 controlling idea，再展开细节。

### 2. 读者优先

每一章都要先回答读者的问题：

- 为什么要做？
- 现有方法有什么问题？
- 本文怎么解决？
- 实验如何证明？
- 还有什么不足？

### 3. 总-分-总

每章建议结构：

- 开头总述本章目标。
- 中间分模块展开。
- 结尾小结本章贡献和下一章承接。

### 4. 先写方法和结果，再写引言

引言需要准确概括全文，因此可以最后再精修。

## 本论文的主线句

建议全文围绕这一句展开：

> 本文在本地低算力和免费 API 限速约束下，设计并实现了一个融合大模型、可迁移 RAG、TTS 与 Live2D 数字人的智能问答平台，并通过分层评估分析其检索、证据、答案和数字人交互效果。

## 写作检查清单

### 摘要

- 是否说明了问题背景？
- 是否说明了系统架构？
- 是否说明了 RAG 方法？
- 是否说明了数字人模块？
- 是否说明了实验结果和不足？
- 是否避免夸大“90% 准确率”？

### 相关工作

- 是否覆盖 RAG 基础论文？
- 是否覆盖 CRAG / DeepNote 等高级 RAG？
- 是否覆盖 RAGAS / ARES / RAGChecker / RAGEval 评估？
- 是否覆盖 AIRI / wLipSync / Live2D？
- 是否说明本文和这些工作的差异？

### 系统设计

- 是否有总体架构图？
- 是否清楚区分前端、后端、RAG、TTS、数字人？
- 是否说明 API？
- 是否说明数据流？

### 系统实现

- 是否写到具体文件或模块？
- 是否解释 PortableRAGV4 的关键步骤？
- 是否解释 AIRI 迁移不是表层引用？
- 是否说明多模型 profile 的必要性？

### 实验评估

- 是否说明数据规模？
- 是否区分代理指标和严格标注指标？
- 是否有负例集？
- 是否分析失败原因？
- 是否写清楚实验限制？

### 总结展望

- 是否总结完成内容？
- 是否诚实写出不足？
- 是否给出合理改进方向？

## 禁忌写法

避免：

- “本文完全复现了某某论文全部方法。”
- “系统准确率达到 90%。”
- “数字人口型完全自然。”
- “RAG 解决了大模型幻觉问题。”

推荐：

- “本文参考并工程化实现了 CRAG 的证据质量评价思想。”
- “代理指标显示检索和证据覆盖较好，但严格答案标注仍暴露不足。”
- “口型同步相比单纯音量开合更自然，但仍未达到商业虚拟主播级别。”
- “RAG 可以降低幻觉风险，但不能完全消除幻觉。”

## 交稿前文件检查

- 论文题目是否与系统实际一致。
- 图表编号是否连续。
- 参考文献是否都在正文引用。
- 英文缩写第一次出现是否解释。
- 所有实验数字是否能在 `experiments/rag_reproduction` 中找到来源。
- API key 是否没有出现在论文截图或附录中。
- 截图中是否没有敏感信息。
