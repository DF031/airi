# 答辩材料

## 3 分钟项目介绍稿

各位老师好，我的毕业设计题目是《融合大模型技术的数字人智能问答平台设计与研究》。本项目主要解决两个问题：第一，如何让大模型基于校园知识库回答规章制度和办事流程类问题，减少事实幻觉；第二，如何将问答系统和数字人交互结合，让系统不仅能回答，还能通过语音、口型和动作进行表达。

系统采用前后端分离架构。后端使用 FastAPI，负责 RAG 检索、大模型调用、记忆管理和 TTS 语音合成；前端使用 React、Vite、PixiJS 和 Live2D，负责问答界面、知识来源展示和数字人表现。大模型默认使用智谱 GLM-4-Flash-250414，TTS 使用 edge-tts。

在 RAG 部分，我没有继续针对固定问题写规则，而是构建了一个无训练、轻量、可迁移的 PortableRAGV4。它通过文档结构化、多路稀疏检索、RRF 融合、证据评分、相邻证据扩展和 MMR 选择来组织证据。实验中，我将评估拆分为检索、证据、答案和安全拒答四层，避免只用单一指标评价系统。

在数字人部分，我参考并迁移了 AIRI 开源项目中的音频 pipeline、wLipSync 口型同步、Live2D motion 管理、自动眨眼、眼球注视和逐模型 profile。平台现在支持多个 Live2D 模型切换、多种中文 TTS 音色、语音播放中断替换和数字人调试面板。

最终系统已经实现从用户提问、RAG 检索、大模型回答、知识来源展示、TTS 播放到数字人口型动作同步的完整闭环。实验也说明，系统在检索和证据覆盖上表现较好，但严格答案准确率和负例拒答仍有改进空间，这也是后续工作的重点。

## 可能被问的问题

### 1. 你的系统和普通 ChatGPT 套壳有什么区别？

回答要点：

- 有本地校园知识库，不依赖大模型参数知识直接回答。
- 有 RAG 检索和知识来源展示。
- 有数字人语音、动作、口型同步。
- 有离线评估和严格标注，不只是调 API。

### 2. 为什么不用训练或微调？

回答要点：

- 毕设目标是本地低成本可运行平台。
- 本地没有高性能 GPU。
- 免费 API 存在限速。
- RAG 更适合频繁更新的校园规章和办事流程，知识更新只需替换文档和索引。

### 3. 为什么不说准确率 90%？

回答要点：

- 代理指标 `usable_proxy_rate` 接近 0.89，但它不是严格答案准确率。
- 严格标注下 positive usable accuracy 约为 0.73。
- 论文中区分了检索命中、证据覆盖、答案可用和负例拒答。
- 这样更严谨，也能说明系统真实瓶颈。

### 4. RAG 复现了哪些前沿思想？

回答要点：

- RAG 基本框架来自 Lewis et al.。
- CRAG 的证据质量评价和纠错思想。
- DeepNote 的 note-centric 多步知识累积思想。
- RAGEval、RAGChecker、RAGAS、ARES 的分层评估思想。
- UltraRAG 的模块化 pipeline 思想。

### 5. 你的 RAG 为什么叫可迁移？

回答要点：

- 不在 Python 代码里写固定校园问题规则。
- 知识库路径、domain profile、数据集和评估配置可替换。
- 检索、融合、证据评分、拒答都是通用算法。
- 当前校园知识库只是一个 domain profile。

### 6. 数字人部分是否只是放了一个 Live2D？

回答要点：

- 不是。
- 做了模型加载、切换、自适应缩放、idle motion、interaction motion。
- 做了 TTS 语音播放队列、替换和中断。
- 做了 wLipSync 音频驱动口型。
- 做了每模型 profile 和调试面板。
- 参考 AIRI 源码进行了模块级迁移。

### 7. 当前系统最大不足是什么？

回答要点：

- 负例拒答能力不足。
- 电话、数字、日期类精确事实召回还不够稳。
- TTS 和口型还达不到商业虚拟主播水准。
- 后续可引入更强结构解析、轻量 reranker、更高质量 TTS 和多知识库迁移实验。

## 答辩演示顺序

1. 打开平台首页。
2. 展示系统状态：GLM、RAG、知识库、TTS。
3. 提问一个校园规章问题。
4. 展示回答和知识来源。
5. 播放语音，展示数字人口型和 Level。
6. 切换 Live2D 模型。
7. 展开调试面板，展示 profile 和动作。
